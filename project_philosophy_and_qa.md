🧠 Project Philosophy & Q&A Summary (Google Music)
이 문서는 개발 과정에서 논의된 주요 아키텍처 결정 사항과 사용자 질문에 대한 답변을 정리한 전략 가이드입니다.
1. 시스템 설계 및 아키텍처 결정
Client-Server 방식 채택 이유: 안드로이드 기기의 리소스(배터리, 발열)를 보호하고, 유튜브의 빈번한 로직 변경에 서버단(yt-dlp)에서 유연하게 대응하기 위함.
하이브리드 저장 전략:
로컬: 서버에서 다운로드 완료 즉시 폰으로 스트리밍 전송하여 데이터 주권 확보.
클라우드: 구글 드라이브(drive.file scope) 연동을 통해 영구 보관 및 기기 간 동기화 지원.
작업 격리 (Task Isolation): 여러 사용자가 동시에 사용할 것을 대비해 task_id 기반의 고유 폴더에서 작업을 수행하도록 설계.
2. 네트워크 및 연결 전략
에뮬레이터 통신: 개발 PC에서 서버 구동 시 10.0.2.2 주소를 통해 호스트 컴퓨터에 접근하는 방식 정립.
외부 접속 (Tailscale): 포트 포워딩의 복잡함과 보안 취약점을 해결하기 위해 P2P VPN인 Tailscale을 도입.
장점: 전 세계 어디서든 고정된 IP(100.x.x.x)로 접속 가능하며, 로컬망에서는 자동 피어링으로 광속 전송 지원.
보안 정책: Android의 Cleartext 통신 허용 설정 및 INTERNET 권한의 계층적 구조 이해.
3. 인프라 및 하드웨어 가이드 (ASUS X502C 서버)
서버 환경: Ubuntu Desktop 기반으로 고성능 yt-dlp 및 FFmpeg 엔진 가동.
물리 설정:
노트북 덮개를 닫아도 서버가 유지되도록 logind.conf 수정 (ignore 설정).
발열 관리를 위해 노트북을 뒤집어 배치(통풍구 상향)하는 실용적 전략 수립.
HDD 장착 기기의 경우 이동 시 데이터 손상 주의 사항 확인.
4. UI/UX 디자인 원칙
상태 기반 UI (State-based UI): 서버가 요리(변환)를 마칠 때까지 '업로드' 버튼을 숨겨 사용자 실수를 원천 차단.
유튜브 스타일 UX:
재생 화면 제스처(좌: 밝기, 우: 볼륨, 더블탭: 5초 이동).
유튜브 믹스 스타일의 접이식 리스트: 접으면 자막, 펼치면 재생 목록 노출.
세션 유지: 탭 이동 시 데이터가 휘발되지 않도록 MainActivity 수준에서 공유 ViewModel 관리.
가독성: 서버는 기계적인 ID로 관리하되, 사용자의 폰에는 실제 영상 제목으로 저장하여 편의성 증대.
5. 주요 기술적 교훈 (Error Patch 핵심)
Kotlin vs Java OptIn: 코틀린 전용 문법 사용 시 반드시 kotlin.OptIn을 임포트해야 함.
Material 3 버전 관리: 최신 문법(HorizontalDivider, ListItem의 headlineContent)과 구형 문법의 차이를 인지하고 프로젝트 버전에 맞춰 고정함.
SHA-1 인증: 구글 로그인의 핵심은 Google Cloud Console에 정확한 패키지명과 디버그 키 지문을 등록하는 것임.
6. 향후 고도화 방향 (Roadmap)
자막 시스템: 서버에서 자막 파일 추출 후 앱에서 실시간 싱크 재생.
미디어 안정성: 0:00 재생 에러 해결 및 Scoped Storage 환경에서의 URI 접근 권한 강화.
백그라운드 서비스: 앱을 닫아도 음악이 끊기지 않는 오디오 세션 구현.
